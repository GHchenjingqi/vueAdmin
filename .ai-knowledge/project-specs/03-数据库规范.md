# 数据库规范

## 表设计

### 命名规范
- **表名**：下划线命名法（snake_case），使用复数形式
  - ✅ 正确：`users`, `roles`, `user_roles`, `dict_data`
  - ❌ 错误：`User`, `user`, `userRoles`, `dictdata`
- **字段名**：下划线命名法
- **主键**：必须使用 `id` 字段，自增整数
- **外键**：格式为 `{table}_id`，如 `user_id`, `role_id`, `dept_id`

### 基础字段
每张表必须包含：
```sql
id INT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
```

### 状态字段
- 使用 `status TINYINT(1) NOT NULL DEFAULT 1 COMMENT '状态 0禁用 1启用'`
- `0` - 禁用/删除
- `1` - 启用/正常

### 排序字段
- 使用 `sort TINYINT(1) UNSIGNED NOT NULL DEFAULT 0 COMMENT '排序'`
- 值越小越靠前

### 软删除
- 如果需要软删除，添加 `deleted_at DATETIME NULL COMMENT '删除时间'`
- 删除时设置为当前时间，查询时过滤 `deleted_at IS NULL`

## 字段类型选择

### ID/主键
- `INT UNSIGNED` - 自增主键，范围足够
- 如果数据量预计很大，可以使用 `BIGINT UNSIGNED`

### 字符串
- `VARCHAR(N)` - 存储较短字符串，`N` 是最大长度
- `TEXT` - 存储较长文本（>255 字符），如简介、内容
- 不要使用 `CHAR` 除非是固定长度字符串（如验证码 key）

### 数字
- `TINYINT(1)` - 布尔/状态（0/1）
- `TINYINT UNSIGNED` - 小范围整数（0-255），如排序
- `INT UNSIGNED` - 普通整数、ID
- `BIGINT UNSIGNED` - 大整数
- `DECIMAL(10,2)` - 货币、精确小数
- 不要用 `FLOAT`/`DOUBLE` 存精确数值（如金额）

### 日期时间
- `DATETIME` - 存储日期时间（推荐，不受时区影响）
- `TIMESTAMP` - 存储时间戳（会受时区影响，不推荐）
- `DATE` - 仅存储日期
- 不需要存 `created_by`, `updated_by` 除非需要审计追踪

## 约束和索引

### 主键
- 每张表必须有主键，使用 `id`

### 唯一约束
- 需要唯一的字段（如用户名、邮箱）要加唯一索引
```sql
UNIQUE KEY `uk_username` (`username`)
```

### 外键约束
- Sequelize 中定义关联关系，但数据库层面不需要强制外键约束
- 应用层保证数据一致性

### 常用索引
- 外键字段必须加索引：`KEY `idx_dept_id` (`dept_id`)`
- 常用查询字段加索引：如 `status`, `created_at`
- 联合索引遵循最左前缀原则

## 字符集和排序规则
- 必须使用 `utf8mb4` 字符集
- 排序规则使用 `utf8mb4_unicode_ci`
```sql
CREATE TABLE `users` (
  ...
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户表';
```

## 关系设计

### 一对一
- 外键放在任意一方，添加唯一索引

### 一对多
- 多方保存 `parent_id`（一方的 ID）
- 例如：部门一对多用户，用户表存 `dept_id`

### 多对多
- 需要中间表保存两个外键
- 例如：用户和角色：`user_roles` 表，存 `user_id` + `role_id`
- 中间表两个外键都要加索引，组合唯一约束：`UNIQUE KEY uk_user_role (user_id, role_id)`

## 迁移

### Sequelize 迁移
- 使用 Umzug 管理迁移
- 迁移文件放在 `server/migrations/`
- 命名格式：`YYYYMMDD_HHMMSS_create_table_name.cjs`
- 迁移只增加不修改已执行的迁移，需要修改则新建迁移

### 种子数据
- 初始数据放在种子文件中
- 开发环境自动执行种子
- 生产环境谨慎执行种子

## 示例

### 正确示例
```sql
CREATE TABLE `articles` (
  `id` INT UNSIGNED NOT NULL PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
  `title` VARCHAR(200) NOT NULL COMMENT '文章标题',
  `content` TEXT NOT NULL COMMENT '文章内容',
  `author_id` INT UNSIGNED NOT NULL COMMENT '作者ID',
  `category_id` INT UNSIGNED NULL COMMENT '分类ID',
  `status` TINYINT(1) NOT NULL DEFAULT 1 COMMENT '状态 0草稿 1发布',
  `sort` TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '排序',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  KEY `idx_author_id` (`author_id`),
  KEY `idx_category_id` (`category_id`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='文章表';
```

## 常见错误

❌ 禁止：
- 使用 camelCase 字段名
- 使用非 `id` 主键名
- 忘记加注释
- 使用错误字符集 `utf8`（只能存 3 字节，Emoji 存不了）
- 过度使用外键约束，增加数据库复杂度
- 在数据库层面存储业务逻辑（存储过程、触发器），全部放在应用层

✅ 推荐：
- 每条字段都加 `COMMENT` 描述
- 合理添加索引
- 保持表结构简单清晰
- 使用一致的命名规范
