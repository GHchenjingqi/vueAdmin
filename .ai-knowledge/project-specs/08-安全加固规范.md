# 安全加固规范

---

## 一、行级数据权限（dataScope）

### 1.1 概述

后端通过 `dataScope` 实现部门级数据隔离。角色配置 `dataScope` 字段：

- `1` = 全部数据
- `2` = 仅本部门
- `3` = 本级及以下部门

### 1.2 核心逻辑（server/utils/dataScope.ts）

```typescript
import { resolveDataScope, applyDataScopeWhere, isTargetInScope } from '@/server/utils/dataScope'

// 解析当前用户的数据范围
const scope = await resolveDataScope(userDeptId, roleIds)
// scope = { dataScope: 2, deptId: 3, deptIds: [3, 5, 6] }

// 将数据范围约束应用到查询 WHERE 条件
applyDataScopeWhere(where, reqUser, queryDeptId)

// 判断目标用户是否在数据范围内
if (isTargetInScope(targetDeptId, reqUser)) {
  // 允许访问
}
```

### 1.3 规则说明

- 多角色取最宽松范围（min），任一角色为 `1` 则整体为全部
- 旧 Token 缺省 `dataScope` 时默认 scope=1（过渡兼容，无锁人风险）
- 跨部门请求返回 403
- "本级及以下"通过 BFS 遍历部门树，返回包含自身的所有下级部门 ID

### 1.4 安全边界

- **列表**：只返回本部门（或下级部门）数据
- **导出**：同列表范围，不泄露其他部门
- **查看详情**：跨部门 403
- **编辑/删除**：跨部门 403

## 二、CSRF 纵深防御

### 2.1 三层防护

| 层 | 机制 | 说明 |
|----|------|------|
| 1 | `SameSite=Strict` Cookie | 跨站点请求自动不携带 Cookie |
| 2 | `originValidator` 中间件 | 非 GET 请求校验 Origin/Referer 白名单 |
| 3 | JWT-in-query 禁用 | SSE 端点不接受 `?token=` 查询参数 |

### 2.2 originValidator 中间件（server/middleware/originValidator.ts）

- 对非 GET 请求校验 `Origin` 头
- 无 Origin 的请求（服务器/爬虫/同源）放行
- Origin 不在白名单内且 Referer 不匹配时 403 拒绝
- 白名单配置：`config.app.allowedOrigins`，支持环境变量 `ALLOWED_ORIGINS` 覆盖

### 2.3 trust proxy

- `app.set('trust proxy', 1)` 确保 Nginx/K8s Ingress 场景下 `req.ip` 获取真实客户端 IP
- 防止限流因代理 IP 失效

## 三、认证安全

### 3.1 JWT 双 Token

- Access Token（短有效期，如 30 分钟）
- Refresh Token（长有效期，如 7 天，HttpOnly Cookie）
- 页面刷新自动静默恢复登录态

### 3.2 Cookie 安全

- `secure` 标记根据 `NODE_ENV` 动态设置
- 生产环境强制 HTTPS Only
- `SameSite=Strict` 防止 CSRF

### 3.3 SSE 一次性票据

- 不再在 URL 中暴露 Access Token
- 改为先获取一次性 ticket 再建立连接

## 四、输入校验

### 4.1 Zod 共享 Schema

- 前后端共享校验规则（`server/shared/schemas/`）
- 通过 `validate()` 中间件在校验路由上生效
- 覆盖登录、密码重置、用户 CRUD 等关键路由

### 4.2 SQL 注入防护

- `userController` 中 `INSERT IGNORE` 原始 SQL 替换为 `UserRole.bulkCreate` + `ignoreDuplicates`
- Sequelize ORM 参数化查询天然防注入

### 4.3 XSS 防护

- `sanitizeSvg` 清洗 SVG 中的 `<script>` 和 `on*` 事件
- 翻译内容为纯文本时使用 `{{ t() }}` 避免 XSS

---

## 五、生产环境安全

### 5.1 错误脱敏

- 500 错误不再暴露内部 `err.message`
- 统一返回通用提示
- 错误分级：409（业务冲突）/ 400（参数错误）/ 500（系统错误）

### 5.2 Helmet 安全头

- CSP（内容安全策略）
- HSTS（HTTP 严格传输安全）
- XSS 过滤
- 其他安全 HTTP 头

### 5.3 请求频率限制

- 基于 IP + 用户名的滑动窗口限流
- 15 分钟内最多 5 次失败
- Redis 多实例支持，自动降级到内存模式

### 5.4 密码安全

- bcryptjs 哈希存储
- `verifyPassword` 不再回退明文比对
